# 节点临时处理（WG + 节点连通）

## 1) 一键止血：修复同机 WG 端口冲突

控制面仓库提供脚本：`vpn-api/scripts/wg-owner-heal.sh`

在每台故障节点执行（会自动扫描本机 `wg-node-*`）：

```bash
sudo bash -c '
set -euo pipefail
curl -fsSL http://127.0.0.1:56700/api/node-setup.sh >/dev/null 2>&1 || true
bash /opt/vpn-api/vpn-api/scripts/wg-owner-heal.sh
'
```

默认策略：
- 仅一个 owner 接口固定 `ListenPort=51820`
- 其余 `wg-node-*` 删除 `ListenPort`（自动端口）

可手动指定 owner：

```bash
sudo bash /opt/vpn-api/vpn-api/scripts/wg-owner-heal.sh --owner wg-node-20
```

## 2) 一键排障：定位 sent>0 / received=0

脚本：`vpn-api/scripts/wg-link-triage.sh`

```bash
sudo bash /opt/vpn-api/vpn-api/scripts/wg-link-triage.sh
```

仅排查某个 peer：

```bash
sudo bash /opt/vpn-api/vpn-api/scripts/wg-link-triage.sh --peer node-20
```

## 3) 结果判读

- 若输出 `[ALERT] ... status=sent_only`：
  - 对端 `wg-quick@wg-node-<本节点ID>` 未启动或配置不一致；
  - 双向安全组/防火墙未放行 UDP 51820；
  - `peer_endpoint / peer_pubkey / allowed_ips` 仍不一致。

- 若 `latest handshake` 持续刷新且 `rx/tx` 双向增长：
  - 链路恢复。

## 4) 10 分钟稳定性验收

脚本：`vpn-api/scripts/wg-e2e-verify.sh`

```bash
sudo bash /opt/vpn-api/vpn-api/scripts/wg-e2e-verify.sh --minutes 10
```

仅验某个 peer：

```bash
sudo bash /opt/vpn-api/vpn-api/scripts/wg-e2e-verify.sh --minutes 10 --peer node-20
```

## 5) 说明

- “同网段不同机器都使用 51820”不是问题；
- 真正冲突是“同一台机器多个 WG 接口同时绑定同一个本地端口”。

## 6) 控制面与节点 agent：何时会下发「最新节点 / 组网」数据

容易误解为「API 服务器没有及时把最新信息推给 agent」。**OpenVPN 实例**仍以按需 `update_config` 为主；**节点间 WireGuard** 已在常见写库路径上增加**自动 `update_wg_config`**（见下 6.3），但 **仅对当前 WebSocket 在线且 agent 声明 `wg_refresh_v1` 的节点** 生效；离线节点仍须在上线后手动「刷新WG」或重拉配置。

### 6.1 Agent WebSocket 心跳

- Agent 周期性发送 `heartbeat`；API 侧主要用于把节点标为在线（`ws_hub.go`）。
- **心跳不会附带隧道列表、不会下发 OpenVPN 全量实例、不会自动 `update_wg_config`。**

### 6.2 OpenVPN 实例（`last-config.json`）

- 通过 WS 消息 `update_config` 推送，典型入口：`pushInstancesConfigToNode`（例如修改实例后），且 **仅当该节点当前已建立 Agent WebSocket** 时才会发送；离线节点不会排队补发。
- 并非每一次写库都会调用推送，取决于具体接口是否显式调用了 `pushInstancesConfigToNode`。

### 6.3 节点间 WireGuard（组网隧道）

- 通过 WS 消息 `update_wg_config` 下发，负载与 **`POST /api/nodes/:id/wg-refresh`（控制台「刷新WG」）** 同源：`BuildTunnelConfigsForNode` 现场快照。
- **自动推送**（对受影响节点逐个尝试发送；不满足「在线 + wg_refresh_v1」则跳过并打日志）已在以下场景接入：
  - **新建节点**：向所有已有对端节点推送（新节点侧通常尚未 WS 在线，依赖 bootstrap）。
  - **`PATCH` 隧道**、**`repair-mesh`**：向隧道两端或本次新建涉及的所有节点推送。
  - **删除节点**：向删除前与该节点有隧道相连的**幸存节点**推送（便于 agent 删掉幽灵 `wg-node-*`）。
  - **`PATCH` 节点且公网 IP 变更**：向该节点在隧道表中的**所有邻居**推送（邻居的 `Endpoint` 依赖对端公网 IP）。
  - **Agent WS `report` 且 `wg_pubkey` 与库中旧值不同**：向**所有邻居**推送，使对端尽快拿到新公钥。
- 仍可随时用手动「刷新WG」补发或排障。
- **离线节点**不会排队补发；上线后需手动刷新或重 bootstrap。旧版 agent 无幽灵 conf 清理时，可手工 `systemctl`/`rm`；新版见 agent `removeStaleWGPeerConfigs`。

### 6.4 与 `bootstrap-node.json` 的关系

- 节点装机/bootstrap 时落盘的 `bootstrap-node.json` 是**当时**的快照；之后若未收到 `update_config` / `update_wg_config`，**磁盘上的 bootstrap 不会随控制面自动改写**。
- Agent 部分逻辑会优先读 `last-config.json` 再回退 bootstrap，与「控制面是否推送」强相关。

### 6.5 运维建议

- 做完**实例**变更后：若节点未自动收到 `update_config`，仍可点控制台相关能力或依赖现有 `pushInstancesConfigToNode` 路径。
- **WG**：在线节点多数情况已自动刷新；变更后若某台仍异常，对该台再执行一次 **「刷新WG」**。
- 长期离线的节点上线后：**重新 bootstrap** 或立刻 **刷新WG**，避免与库漂移。
