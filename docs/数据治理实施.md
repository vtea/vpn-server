# 数据治理实施（SQLite + PostgreSQL）

本文说明：**如何落地**「备份 → 体检 → 修复 → 下发 → 验证」全流程，并与当前仓库能力对齐（`repair-mesh`、自动 `update_wg_config`、agent 写 conf 净化等）。

---

## 一、两套体检 SQL（已实现）

| 引擎 | 文件 |
|------|------|
| SQLite | `vpn-api/scripts/db-health-check.sqlite.sql` |
| PostgreSQL | `vpn-api/scripts/db-health-check.postgres.sql` |

检查项（两文件逻辑一致，仅可疑字符检测语法随方言略有差异）：

1. **孤儿隧道**：`tunnels.node_a/node_b` 在 `nodes` 中不存在  
2. **孤儿实例**：`instances.node_id` 无对应节点  
3. **孤儿 node_segments**  
4. **`wg_public_key` 为空的节点**  
5. **`tunnels.subnet` 重复**  
6. **可疑 `nodes.id`**（含 `/`、`\`、换行、`..` 等，与 agent 路径安全策略冲突）  
7. **孤儿 tunnel_metrics**  
8. **孤儿 user_grants**（`instance_id` 不存在）

### 1.1 SQLite 执行方式

**不要**对正在由 `vpn-api` 打开的同一 `.db` 直接长时间跑查询（易 `database is locked`）。**先复制真实库再体检**。

**重要**：`sqlite3 某个.db` 若该文件**尚不存在**，会**新建空库** → `00_schema_sanity` 里 `total_tables=0`，后面全部 **`no such table`**。必须先 **`cp -a` 带表结构的那个 `vpn.db`**，再 `ls` 确认副本非 0 字节。

下面分两种目录布局（不要混用）。

#### A) Git 仓库 `vpn-server/`（含子目录 `vpn-api/`、`docs/`）

当前目录为**仓库根**（能看到 `vpn-api` 与 `docs` 两个文件夹）：

| 场景 | 真实库路径（相对仓库根） |
|------|---------------------------|
| 本地默认（`DB_PATH=./vpn.db` 且进程 cwd 为 `vpn-api`） | `vpn-api/vpn.db` |
| 本仓库 Docker Compose | `vpn-api/docker-data/data/vpn.db` |

```bash
cp -a vpn-api/vpn.db vpn-api/vpn-healthcheck.db
ls -la vpn-api/vpn-healthcheck.db
sqlite3 vpn-api/vpn-healthcheck.db ".read vpn-api/scripts/db-health-check.sqlite.sql"
```

#### B) 控制面安装目录（`deploy-control-plane.sh` / `package-linux.sh` 默认 `INSTALL_DIR=/opt/vpn-api`）

服务器上常见结构是 **`/opt/vpn-api/data/vpn.db`** 为真实库，**`/opt/vpn-api/scripts/`** 为脚本（你当前 `ls /opt/vpn-api` 即属此类）。此时**不要**假设还有一层 `vpn-api/vpn.db`；副本应来自 **`data/vpn.db`**。

```bash
cd /opt/vpn-api
cp -a data/vpn.db ./vpn-healthcheck.db
ls -la data/vpn.db vpn-healthcheck.db
sqlite3 vpn-healthcheck.db ".read scripts/db-health-check.sqlite.sql"
```

若你习惯在 **`/opt`** 下操作（与 `ls` 一致），等价于：

```bash
cp -a /opt/vpn-api/data/vpn.db /opt/vpn-api/vpn-healthcheck.db
sqlite3 /opt/vpn-api/vpn-healthcheck.db ".read /opt/vpn-api/scripts/db-health-check.sqlite.sql"
```

脚本第 **0** 节 `00_schema_sanity`：`nodes_table_present`、`tunnels_table_present` 表示 **是否存在名为 `nodes` / `tunnels` 的表**（取值只有 **0 或 1**），**不是**「库里有多少个节点」。二者都应为 **1**；若为 **0**，说明副本仍是空库或拷错文件，先不要解读后面各节。第 **01～08** 节若**完全没有输出行**，表示该项检查通过（不是脚本坏了）。

Windows PowerShell（**仅适用于本机 Git 布局 A**）：

```powershell
Copy-Item -Force vpn-api\vpn.db vpn-api\vpn-healthcheck.db
sqlite3 vpn-api\vpn-healthcheck.db ".read vpn-api/scripts/db-health-check.sqlite.sql"
```

等价（仓库根）：

```bash
sqlite3 vpn-api/vpn-healthcheck.db < vpn-api/scripts/db-health-check.sqlite.sql
```

（脚本内含 `.headers on` / `.mode column`。）

### 1.2 PostgreSQL 执行方式

连接串来自你部署时的环境变量（与 `vpn-api/cmd/api/main.go` 中 `DB_DRIVER=postgres` + `DB_PATH` 为 DSN 一致）。**仓库根**下执行体检脚本：

```bash
psql "$DB_PATH" -f vpn-api/scripts/db-health-check.postgres.sql
```

若你习惯用 `DATABASE_URL` 命名，可先 `export DATABASE_URL=...` 再 `psql "$DATABASE_URL" -f vpn-api/scripts/db-health-check.postgres.sql`。

## 二、最佳方案：具体怎么实现（按顺序做）

### 阶段 0：基线

- 控制面、**vpn-agent**、管理端版本与提交号一致（避免一半新一半旧）。  
- 确认 `DB_DRIVER`：`sqlite` 或 `postgres`，与 `DB_PATH`/连接串一致（见 `vpn-api/cmd/api/main.go` 中 `openDB`）。

### 阶段 1：备份（必做）

**SQLite**

- **Git 布局 A（仓库根）**：与 §1.1 相同，源可为 `vpn-api/vpn.db` 或 `vpn-api/docker-data/data/vpn.db`。  
- **安装布局 B（`/opt/vpn-api`）**：源为 **`/opt/vpn-api/data/vpn.db`**，备份到 **`/opt/vpn-api/backups/`**（与 `deploy-control-plane.sh` 一致）。

```bash
# 布局 A：仓库根
mkdir -p vpn-api/docker-data/backups
cp -a vpn-api/vpn.db "vpn-api/docker-data/backups/vpn-$(date +%Y%m%d%H%M).db"

# 布局 B：安装目录
mkdir -p /opt/vpn-api/backups
cp -a /opt/vpn-api/data/vpn.db "/opt/vpn-api/backups/vpn-$(date +%Y%m%d%H%M).db"
```

若当前用的是 Compose 数据目录下的库，则把源文件改为 `vpn-api/docker-data/data/vpn.db`，目标仍建议落在同仓库的 `vpn-api/docker-data/backups/`。

**PostgreSQL**

```bash
mkdir -p vpn-api/docker-data/backups
pg_dump "$DB_PATH" --format=custom --file="vpn-api/docker-data/backups/vpn-$(date +%Y%m%d%H%M).dump"
```

（`DB_PATH` 与运行 `vpn-api` 时配置的 Postgres DSN 相同；备份目录与 `vpn-api/docker-compose.yml` 中 `./docker-data/backups` 挂载一致，便于与 Docker 部署统一。）

### 阶段 2：只读体检

按上一节执行对应 SQL。  
**解读**：任一查询有行输出 → 记为待处理项；全空为理想状态。

### 阶段 3：分类修复（实现方式）

原则：**能走 API 就不手写 SQL 改业务表**；**禁止直接改 `nodes.id`**。

| 问题类型 | 推荐实现方式 |
|----------|----------------|
| 孤儿隧道（误删节点导致） | 能恢复节点则恢复；否则 **删除孤儿 `tunnels` 行**（需 SQL 或临时管理工具），再 `POST /api/tunnels/repair-mesh`，最后依赖自动 WG 推送或批量 `wg-refresh`。 |
| 孤儿实例 / node_segments | 通常与删节点不彻底有关：用控制台 **删节点** 走官方逻辑；若只剩孤儿行，可用 SQL `DELETE` 孤儿行（**先备份**）。 |
| `wg_public_key` 空 | 在节点上保证 `/etc/wireguard/publickey` 正确，**agent 连上后 report**；或修正库后触发邻居 **自动 WG 推送**（见 `ws_hub` report 与 handlers）。 |
| `subnet` 重复 / IP 不在 /30 | 使用 **`PATCH /api/tunnels/:id`**（走 `ValidateTunnelWGFields`）；无法修则删该隧道再 **`repair-mesh`**。 |
| 可疑 `nodes.id` | **不要 UPDATE 改 id**。实现上只有：**删节点 + 新建节点**（新 `node_number`/`id`），再 `repair-mesh`，各节点 **bootstrap / 刷新 WG**。 |
| 孤儿 metrics / grants | 维护 SQL `DELETE FROM tunnel_metrics WHERE tunnel_id NOT IN (...)` 等；或保留不动仅影响历史曲线，按业务决定。 |

### 阶段 4：拓扑补全

```http
POST /api/tunnels/repair-mesh
Authorization: Bearer <具备 tunnels 权限的 JWT>
```

作用：按当前 `nodes` 全量补缺失隧道；服务端会对涉及节点 **自动下发 `update_wg_config`**（在线且支持 `wg_refresh_v1` 的 agent）。

### 阶段 5：下发与节点对齐

1. **在线节点**：依赖自动推送；仍异常则控制台对该节点 **「刷新WG」** 或 `POST /api/nodes/:id/wg-refresh`。  
2. **离线节点**：上线后立刻 **刷新 WG** 或重新 **bootstrap**（`bootstrap-node.json` 不会随 DB 自动更新）。  
3. **批量**：可用 `vpn-api/scripts/wg-gate-check.sh`（JWT + API）对多台串行刷新。

### 阶段 6：验证

- 控制台隧道健康 / 审计里 `wg_refresh_auto`、`wg_refresh`。  
- 节点：`sudo wg show all`；对端隧道地址 **双向 ping**。  
- 文档：`docs/节点临时处理.md` 中脚本（`wg-e2e-verify.sh` 等）。

### 阶段 7：固化（可选「实现」为日常）

- 把 **阶段 1–2** 做成 cron（每周）+ 告警（任一体检有结果则通知）。  
- 变更窗口内：**先备份 → 再跑体检 → 再改库/API**。

## 三、与代码能力的对应关系

| 能力 | 代码位置（概念） |
|------|------------------|
| 隧道校验 | `service.ValidateTunnelWGFields` |
| 全互联补边 | `POST /tunnels/repair-mesh` → `EnsureFullMeshTunnels` |

## 四、最短「一键心智模型」

**备份 → 跑对应 SQL 体检 → 按上表用 API/SQL 修 → `repair-mesh` → 在线自动刷新 WG、离线手工刷 → `wg show` 验收。**

若你希望把「孤儿行 DELETE」也做成**带 dry-run 的 shell 脚本**（先 `SELECT` 再确认 `DELETE`），可在后续迭代里加到 `vpn-api/scripts/`。
