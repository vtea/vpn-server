# 节点安装命令：局域网/公网双地址 + 脚本检测重复部署

## 需求

1. **管理端生成安装命令时**，同时提供 **局域网访问控制面的部署命令** 与 **公网（或域名）访问的部署命令**，便于管理员按场景复制（内网装机用 LAN，跨网/公网用 WAN）。
2. **`node-setup.sh`** 需 **检测本机是否已部署过**（例如已存在 `/etc/vpn-agent/agent.json` 或 `bootstrap-node.json` 等标记）；若已部署，**再次执行**时要有明确策略（提示 / 覆盖 / 仅修复），避免静默破坏或无法区分新旧安装。

---

## 现状（代码只读结论）

| 项 | 现状 |
|----|------|
| 部署命令 | [`CreateNode`](openvpn-install-master/vpn-api/internal/api/handlers.go) 仅使用单一 `apiURL := h.externalURL`（来自配置 [`EXTERNAL_URL`](openvpn-install-master/vpn-api/internal/config/config.go)），生成一条 `deploy_command`。 |
| 脚本入口 | [`vpn-api/scripts/node-setup.sh`](openvpn-install-master/vpn-api/scripts/node-setup.sh)，参数 `--api-url`、`--token`、`--apply`。 |
| 本机状态 | 成功部署后会写入 `/etc/vpn-agent/bootstrap-node.json`、`/etc/vpn-agent/agent.json` 等；**脚本开头未检测「已安装」即继续覆盖**。 |
| 预检端口 | UDP 56710–56713、56720 被占用仅 **警告**；若为本节点已起的 OpenVPN/WG，属**重复部署信号**。 |
| Agent 注册 | [`AgentRegister`](openvpn-install-master/vpn-api/internal/api/handlers.go) 每次请求仍会更新节点并把 bootstrap token 标为已使用；**同一 token 多次 register 当前仍会成功**（需在产品上定义是否允许、是否应轮换 token）。 |

## 方案 A：双地址部署命令（控制面配置 + API 响应）

1. **配置扩展**（命名可再定稿）：  
   - 保留 **`EXTERNAL_URL`**：对外/公网或主入口（与现有行为一致）。  
   - 新增可选 **`EXTERNAL_URL_LAN`**（或 `DEPLOY_URL_LAN`）：仅用于生成「内网装机」命令；未设置时前端/API 只展示一条命令。

2. **Handler / 响应字段**（`CreateNode` 及若有的「复制部署命令」接口）：  
   - `deploy_command`：基于 `EXTERNAL_URL`（或约定主 URL）。  
   - `deploy_command_lan`：仅当 `EXTERNAL_URL_LAN` 非空时生成（`curl … | bash -s -- --api-url <LAN> --token …`）。  
   - `script_url` / `script_url_lan` 同理，便于文档与离线脚本说明。

3. **管理台**：节点创建成功页 / 节点详情中 **两个可复制块**（LAN / 公网），并简短说明「内网节点请用 LAN；需经公网可达请用公网地址」。

4. **文档**：[`docs/build-and-run.md`](openvpn-install-master/docs/build-and-run.md)、[`vpn-api/README.md`](openvpn-install-master/vpn-api/README.md) 增加环境变量说明。

## 方案 B：`node-setup.sh` — 交互式「安装 / 卸载」与已部署分支（修订，2025-04）

### 入口行为（`--apply` 且进入主流程时）

1. **首屏菜单（需 TTY；无 TTY 时见文末非交互）**  
   - 提示选择：**`1) 安装`** 或 **`2) 卸载`**（文案可用中文）。

2. **若选择「卸载」**  
   - 执行 **卸载流程**：停止 vpn-agent、OpenVPN、WireGuard 等相关 systemd 单元；删除或备份 `/etc/vpn-agent`；按脚本既有布局清理 `/etc/openvpn/server` 下本脚本创建的内容（需列出清单，避免误删用户自建目录）；可选清理 iptables/ipset 规则（若可安全识别为本脚本写入）。  
   - 结束后提示完成并 **exit 0**。

3. **若选择「安装」**  
   - 先执行 **是否已部署检测**（强标记优先）：存在 `/etc/vpn-agent/agent.json` 和/或 `bootstrap-node.json` 等即视为已部署。  
   - **若未部署**：进入现有 **环境预检** → Step 1–9 安装流程（与现逻辑衔接）。  
   - **若已部署**：  
     - **二次确认**：是否 **卸载并清空本机已部署数据** 后 **重新安装**？（是/否）  
     - **若选「是」**：先调用与「卸载」相同的 **清空逻辑**（或内联 `do_uninstall`），再 **继续** 执行预检与完整安装。  
     - **若选「否」**：**不继续安装**；打印 **当前部署信息**，至少包括：  
       - **当前连接的 API**：从 `/etc/vpn-agent/agent.json` 读取 **`api_url`** 字段（脚本已写入 `${API_URL}`，见 `vpn-api/scripts/node-setup.sh` 中 `agent.json` 生成段）；  
       - 可选：`node_id`、`node_number`（来自 `bootstrap-node.json` 或 `jq` 解析）；  
       - 一句说明：「未修改本机配置；若需换控制面或重装，请先选择卸载清空或重新运行并选择清空重装。」  
     - 然后 **exit 0**（信息性退出，非错误）。

### 检测依据（与上配合）

- **强标记**：`/etc/vpn-agent/agent.json`（含 **`api_url`**，供「当前连哪个控制面」展示）。  
- **辅助**：`bootstrap-node.json`；端口占用与已部署一致性可作补充判断。

### 与端口预检的关系

- 在「安装」路径中，预检仍可在 **清空重装** 之后执行；若清空后端口释放，告警应消失。  
- **仅查看信息**（用户选不清空）时 **不必** 再跑全量预检，避免误导性错误。

### 非交互 / 自动化（实现时必带）

- 无 TTY 或传入 **`--non-interactive`** / **`--yes`** 时：需定义默认行为（例如：**已部署则失败退出** 或 **必须同时传 `--force-reinstall`**），并在 `usage` 中写明，避免 CI 挂死在读输入上。

### 与旧「仅 CLI 标志」方案的关系

- 原 **`--repair` / `--force-reinstall`** 可作为 **非交互** 场景的等价物保留；交互菜单为 **默认推荐路径**。  
- **Bootstrap token 策略**仍见下文「后端配合」。

### 后端配合（可选）

- **Bootstrap token 一次性**：若产品要求「第二次部署必须换新 token」，需在 `AgentRegister` 对 `bt.Used == true` 返回 **401** 并引导在管理台 **重新生成 bootstrap**；与「repair / 仅展示信息」路径需统一设计。  
- 当前实现会多次 register 成功，计划实现阶段需 **显式选择**：允许 repair 用同一 token **或** 强制新 token。

## 执行清单（实现阶段）

1. **config**：增加 `EXTERNAL_URL_LAN`（可选）；`Load()` 与文档同步。  
2. **handlers**：`CreateNode`（及任何返回 deploy 命令处）输出双命令与双 `script_url`。  
3. **vpn-admin-web**：节点详情/创建结果展示两段可复制命令。  
4. **node-setup.sh**：实现 **安装/卸载菜单**；安装路径下 **已部署 → 是否清空重装** 分支；选否时 **输出 `agent.json` 中 `api_url` 等当前部署信息**；抽取 **`do_uninstall`/`do_purge`** 供卸载与重装前清空复用；补充 **非交互** 策略与 `usage`。  
5. **（可选）** `AgentRegister` 与 token 策略对齐产品决策。  
6. **文档**：安装指南中「内网 vs 公网」与「重复执行脚本」说明。
